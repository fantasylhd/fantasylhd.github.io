设计模式之单例设计模式

1. 单例的定义

单例模式 ( Singleton Pattern )是指确保一个类在任何情况下都绝对只有一个实例，并提供一个全局访问点。单例模式是创建型模式。

2. 单例的优点

  1. 节省内存和计算

  2. 保证结果正确

  3. 方便管理

3. 单例的缺点

  1. 单例对OOP特性的支持不友好
  2. 单例会隐藏类之间的依赖关系
  3. 单例对代码的扩展性不友好
  4. 单例不支持有参数的构造函数
  5. 单例对代码的可测试性不友好

4. 单例的应用场景

  1. 全局信息类：有些数据在系统中只应该保存一份，就比较适合设计为单例类。比如，系统的配置信息类、记录网站的访问次数、连接池类、ID生成器类。

  2. 无状态的工具类︰比如日志工具类，不管是在哪里使用，我们需要的只是它帮我们记录日志信息，除此之外，并不需要在它的实例对象上存储任何状态，这时候我们就只需要一个实例对象即可。

  3. 除此之外，我们还可以使用单例解决资源访问冲突的问题，还需要加上对象锁的形式。

5. 单例的重点

  1. 私有构造器
  2. 线程安全
  3. 延迟加载
  4. 序列化和反序列化安全
  5. 反射

6. 单例的实现

6.1 饿汉式

//利用静态代码块的方式实现饿汉式
public class HungrySingleton implements Serializable,Cloneable{

    private final static HungrySingleton hungrySingleton;

    static{
        hungrySingleton = new HungrySingleton();
    }

    private HungrySingleton(){
    }

    public static HungrySingleton getInstance(){
        return hungrySingleton;
    }

}

public class HungrySingleton implements Serializable,Cloneable{

    private final static HungrySingleton hungrySingleton = new HungrySingleton();

    private HungrySingleton(){
    }

    public static HungrySingleton getInstance(){
        return hungrySingleton;
    } 
}

6.1.1 实现原理

  1. 利用类加载过程来实例化对象来保证线程的安全
  2. 使用final关键字来保证引用不被修改
  3. 静态代码块在类加载阶段也会被执行

6.1.2 优缺点

优点：

  1. 写法简单、易于理解
  2. 实现了线程安全、不加任何其他锁、效率高

缺点：

  1. 没有实现懒加载、不用到这个对象可能会造成内存的浪费

不过这个缺点不是一定的、下面引用别人的一段说法：

有人觉得这种实现方式不好，因为不支持延迟加载，如果实例占用资源多(比如占用内存多)或初始化耗时长(比如需要加载各种配置文件)，提前初始化实例是一种浪费资源的行为。最好的方法应该在用到的时候再去初始化。不过，我个人并不认同这样的观点。

如果初始化耗时长，那我们最好不要等到真正要用它的时候，才去执行这个耗时长的初始化过程，这会影响到系统的性能（比如，在响应客户端接口请求的时候，做这个初始化操作，会导致此请求的响应时间变长，甚至超时)。采用饿汉式实现方式，将耗时的初始化操作，提前到程序启动的时候完成，这样就能避免在程序运行的时候，再去初始化导致的性能问题。

如果实例占用资源多，按照 fail-fast的设计原则(有问题及早暴露），那我们也希望在程序启动时就将这个实例初始化好。如果资源不够，就会在程序启动的时候触发报错（比如Java中的PermGen Space OOM)，我们可以立即去修复。这样也能避免在程序运行一段时间后，突然因为初始化这个实例占用资源过多，导致系统崩溃，影响系统的可用性。

6.2 懒汉式

public class LazySingleton {

    private static LazySingleton lazySingleton = null;

    private LazySingleton(){
    }

    public synchronized static LazySingleton getInstance(){
        if(lazySingleton == null){
            lazySingleton = new LazySingleton();
        }
        return lazySingleton;
    }
}

6.2.1 实现原理

  1. 首先不去创建出对象、而是在调用方法时才创建对象
  2. 使用synchronized锁方法、来达到线程安全

6.2.2 优缺点

优点：

  1. 在饿汉式的基础上实现了懒加载

缺点：

  1. 多线程同时访问的效果很差、相当于是串行。一次的锁竞争之后、就可以直接返回对象了。

6.3 双重检查锁

public class LazyDoubleCheckSingleton {

    private volatile static LazyDoubleCheckSingleton lazyDoubleCheckSingleton = null;

    private LazyDoubleCheckSingleton(){
    }

    public static LazyDoubleCheckSingleton getInstance(){
        if(lazyDoubleCheckSingleton == null){
            synchronized (LazyDoubleCheckSingleton.class){
                if(lazyDoubleCheckSingleton == null){
                    lazyDoubleCheckSingleton = new LazyDoubleCheckSingleton();
                    //1.分配内存给这个对象
//                  //3.设置lazyDoubleCheckSingleton 指向刚分配的内存地址
                    //2.初始化对象
//                    intra-thread semantics
//                    ---------------//3.设置lazyDoubleCheckSingleton 指向刚分配的内存地址
                }
            }
        }

        return lazyDoubleCheckSingleton;
    }
}

6.3.1 实现原理

  1. 第一个if来规避无效的加锁操作，第二个if来保证线程的安全(线程同时进入第一个if的情况)
  2. 使用synchronized来保证线程安全
  3. 使用volatile来保证发生了指令重排序之后、线程访问到未初始化完成的对象的情况 (禁止指令重排序)

6.3.2 优缺点

优点：

  1. 一次锁竞争之后就不用加锁了，效率很高

缺点：

  1. 还是需要加锁
  2. 不容易理解

6.4 静态内部类

public class StaticInnerClassSingleton {

    private static class InnerClass{
        private final static StaticInnerClassSingleton staticInnerClassSingleton = new StaticInnerClassSingleton();
    }
    //方法上声明static 是为了共享单例，便于调用
    public static StaticInnerClassSingleton getInstance(){
        return InnerClass.staticInnerClassSingleton;
    }

    private StaticInnerClassSingleton(){
    }
}

6.4.1 实现原理

  1. 静态内部类使用的时候才会加载、达到了懒加载的效果
  2. 利用类加载过程来实例化对象来保证线程的安全

6.4.2 优缺点

优点：

  1. 不需要加锁
  2. 达到了懒加载的效果

缺点：

没有很明显的缺点

6.5 枚举

public enum EnumInstance {

    INSTANCE;

    public static EnumInstance getInstance(){
        return INSTANCE;
    }
}

6.5.1 实现原理

  1. 利用了枚举的特性

6.5.2 优缺点

优点：

  1. 线程安全，并且不会被反射和序列化破坏

缺点：

  1. 没有实现懒加载
  2. 语义不是很明确

6.6 容器式单例

public class ContainerSingleton {

    public static Map<String,Object> map = new ConcurrentHashMap<>();

    public static Object getInstance(String key){
        if(key!=null){
            if(!map.containsKey(key)){
                synchronized (map) {
                    Object single = null;
                    if (!map.containsKey(key)) {
                        single = new Object();
                        map.put(key, single);
                        return single;
                    }
                }
            }
        }
        return map.get(key);
    }

    private ContainerSingleton(){

    }
}

6.6.1 实现原理

  1. 利用ConcurrentHashMap，构造一个key,存储单例对象
  2. 利用synchronized实现线程安全，采用双重检查来保证效率

6.6.2 优缺点

优点：

  1. 便于统一管理单例对象
  2. 实现了延迟加载

缺点：

  1. 达到线程安全需要加锁，影响效率

6.7 ThreadLocal线程单例

public class ThreadLocalInstance {

    private static final ThreadLocal<ThreadLocalInstance> threadLocal = new ThreadLocal<ThreadLocalInstance>(){
        @Override
        protected ThreadLocalInstance initialValue() {
            return new ThreadLocalInstance();
        }
    };

    private ThreadLocalInstance(){

    }

    public ThreadLocalInstance getInstance(){
        return  threadLocal.get();
    }
}

6.7.1 实现原理

  1. 利用ThreadLocal加类加载过程来初始化对象，每个线程都有一个ThreadLocalMap成员来存储自己的对象

6.7.2 优缺点

  1. 不能创建全局唯一的单例类

7. 反射破坏单例模式

public class LazyInnerClassSingletonTest {

    public static void main(String[] args) {
        try{
            //很无聊的情况下，进行破坏
            Class clazz = LazySingleton.class;
            //通过反射拿到私有的构造方法
            Constructor c = clazz.getDeclaredConstructor(null);
            //强制访问
            c.setAccessible(true);
            //暴力初始化
            Object o1 = c.newInstance();

            Object o2 = LazySingleton.getInstance();
            System.out.println(o1 == o2);
        }catch (Exception e){
            e.printStackTrace();
        }
    }
}

7.1 解决方案

private HungrySingleton(){
    if(hungrySingleton != null){
        throw new RuntimeException("不允许重复创建对象");
    }
}

在构造方法中进行判断：一旦出现多次重复创建，则直接抛出异常。对于懒汉式来说，如果反射先执行的话会创建出2个对象

8. 序列化破坏单例模式

public class SeriableSingletonTest {

    public static void main(String[] args) {

        HungrySingleton s1 = null;
        HungrySingleton s2 = HungrySingleton.getInstance();
        FileOutputStream fos = null;

        try {

            fos = new FileOutputStream("singleton_file");
            ObjectOutputStream oos = new ObjectOutputStream(fos);
            oos.writeObject(s2);
            oos.flush();
            oos.close();

            FileInputStream fis = new FileInputStream("singleton_file");
            ObjectInputStream ois = new ObjectInputStream(fis);
            s1 = (HungrySingleton)ois.readObject();
            ois.close();

            System.out.println(s1);
            System.out.println(s2);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

8.1 解决方案

private Object readResolve(){
    return hungrySingleton;
}

只需要增加 readResolve()方法将创建好的对象返回即可，原理：增加了这个方法之后，序列化的内部会反射调用这个方法得到对象，但是，同时内部也反射创建了一个对象，只不过增加了这方法之后，就没有被返回了
